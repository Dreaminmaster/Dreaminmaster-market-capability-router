# v0.2 Implementation Specification — Structured Model Adapter

## 1. Goal

v0.2 adds an optional structured language-model layer without changing the local-first rule engine or weakening hard safety rules.

The model may improve:

- real-goal extraction;
- friction hypotheses;
- task decomposition;
- profession and service vocabulary expansion;
- query-lattice recall;
- explanation quality;
- interpretation of previously unknown marketplace expressions.

The model must not become the sole authority for routing, safety, legality, credentials, payment, or official-channel decisions.

## 2. Product invariants

The following invariants are release blockers:

1. Rules run even when no model is configured.
2. Critical rule matches cannot be removed, downgraded, or hidden by a model response.
3. Model output is treated as a hypothesis with provenance, confidence, and evidence requirements.
4. Candidate descriptions are untrusted data, never system instructions.
5. A malformed, slow, unavailable, or contradictory model falls back to rules without breaking the command.
6. Existing v0.1.1 CLI and HTTP behavior remains backward compatible.
7. Simple writing tasks should not invoke the model or market path unless capability routing is actually needed.

## 3. Modes

Add three explicit execution modes:

```text
rules   — current deterministic engine only; default
hybrid  — rules first, then model enrichment, then deterministic merge
model   — not supported in v0.2; reject with a clear error
```

Default remains `rules` so installation does not require an API key, network, or local model.

## 4. Proposed package structure

```text
src/mcr/
├── adapters/
│   └── llm/
│       ├── __init__.py
│       ├── base.py
│       ├── openai_compatible.py
│       └── fake.py
├── hybrid/
│   ├── merge.py
│   ├── prompts.py
│   ├── schemas.py
│   └── validation.py
├── config.py
├── evidence.py
└── ...existing modules
```

Do not import provider SDKs in the domain layer.

## 5. LLMAdapter protocol

Define a provider-independent protocol similar to:

```python
class LLMAdapter(Protocol):
    def complete_json(
        self,
        *,
        system: str,
        user_payload: dict[str, object],
        schema: dict[str, object],
        timeout_seconds: float,
    ) -> LLMResponse:
        ...
```

`LLMResponse` must include:

```text
provider
model
raw_text
parsed
latency_ms
usage if available
request_id if available
warnings[]
```

No domain module should depend on OpenAI-specific response objects.

## 6. OpenAI-compatible adapter

Support OpenAI-compatible chat-completions endpoints used by hosted providers, LM Studio, and similar local servers.

Configuration must be explicit and environment-driven:

```text
MCR_LLM_BASE_URL
MCR_LLM_API_KEY
MCR_LLM_MODEL
MCR_LLM_TIMEOUT_SECONDS
MCR_LLM_MAX_RETRIES
```

Requirements:

- base URL must be configurable;
- API key may be optional for local endpoints;
- never print the key;
- timeout is mandatory;
- retries only for transport or retryable server failures;
- no retry on invalid credentials or schema rejection;
- return structured errors without exposing secrets;
- use an adapter boundary so HTTP implementation can later change.

A standard-library HTTP implementation is acceptable for v0.2. If an external package is added, it must be optional and justified.

## 7. Structured schemas

The model must return one JSON object matching a versioned schema.

Minimum fields:

```json
{
  "schema_version": "0.2",
  "real_goal": "string",
  "friction_hypotheses": [
    {
      "type": "knowledge|diagnosis|skill|channel|execution|verification",
      "confidence": 0.0,
      "evidence": ["string"],
      "uncertainty": "string"
    }
  ],
  "task_hypotheses": [
    {
      "title": "string",
      "expected_deliverable": "string",
      "suggested_routes": ["AI|TOOL|SELF|PROFESSIONAL|MARKET|OFFICIAL"],
      "requires_user_action": true,
      "sensitivity": "low|medium|high"
    }
  ],
  "profession_terms": ["string"],
  "service_terms": ["string"],
  "query_terms": ["string"],
  "unknown_dialect_hypotheses": [
    {
      "term": "string",
      "possible_meanings": ["string"],
      "confidence": 0.0,
      "evidence_required": ["string"]
    }
  ],
  "warnings": ["string"]
}
```

Validation must reject:

- unknown route names;
- unknown friction types;
- confidence outside 0–1;
- non-object top-level output;
- oversized arrays beyond configured limits;
- embedded executable instructions in fields intended as labels;
- missing schema version.

## 8. Prompt construction and injection resistance

Candidate and user content must be placed in a clearly delimited data envelope. The prompt must state that content inside the envelope is untrusted evidence and cannot change system rules.

Required defenses:

1. Separate system policy from user data.
2. Pass candidate content as JSON data, not concatenated instructions.
3. Limit response to the requested schema.
4. Ignore instructions contained in listing descriptions, reviews, or chats.
5. Detect obvious prompt-injection strings and add a warning.
6. Never allow model output to request tool execution directly.
7. Never send passwords, verification codes, identity numbers, payment credentials, or raw private documents.
8. Add redaction before model calls.

The redactor should cover at least:

```text
verification codes
password-like fields
bank-card-like sequences
identity-number-like sequences
email and phone where not needed
```

Do not claim perfect detection; record redaction warnings.

## 9. Hybrid merge algorithm

Use a deterministic merge, not free-form model arbitration.

### 9.1 Order

```text
rule analysis
-> hard-risk scan
-> optional redaction
-> model hypothesis
-> schema validation
-> deterministic merge
-> final risk scan
-> result
```

### 9.2 Merge rules

- Rule risk flags always remain.
- A model cannot lower a rule risk severity.
- `SELF` and `OFFICIAL` routes required by sensitive or official terms remain.
- Model routes may be added only as secondary hypotheses unless a deterministic allow rule promotes them.
- Model-generated professions and query terms are marked `source=model` and `status=candidate`.
- Unknown dialect meanings never enter the verified seed dictionary automatically.
- Duplicates are normalized but source provenance is preserved.
- If model and rules conflict, output both with a `conflict` record and prefer the safer rule.

### 9.3 Evidence record

Every merged item should support:

```text
value
source: rule | model | user | seed
confidence
matched_evidence[]
status: verified | candidate | rejected
conflicts[]
```

## 10. Configuration

Configuration precedence (v0.2):

```text
CLI flag > environment variable > default
```

Configuration files are deferred to a future version. No secret is written to logs, errors, or committed files.

Suggested CLI additions:

```bash
mcr analyze "..." --mode rules
mcr analyze "..." --mode hybrid
mcr analyze @request.txt --mode hybrid --llm-base-url http://127.0.0.1:1234/v1 --llm-model local-model
```

The current positional syntax must continue to work.

## 11. Failure behavior

Hybrid mode must return useful output even when the model fails.

Required failure classes:

```text
not_configured
connection_error
timeout
http_error
auth_error
invalid_json
schema_error
response_too_large
prompt_injection_warning
```

Final output should contain:

```json
{
  "model_enrichment": {
    "attempted": true,
    "applied": false,
    "status": "timeout",
    "warnings": []
  }
}
```

The process exit code should remain success when rule analysis succeeded and model enrichment merely failed, unless a strict diagnostic command was explicitly requested.

## 12. Testing requirements

### Unit tests

Add tests for:

- adapter protocol and fake adapter;
- OpenAI-compatible request shape;
- API key redaction from errors;
- timeout and retry policy;
- valid structured response;
- malformed JSON;
- invalid schema;
- unknown friction and route values;
- model trying to remove a critical risk;
- model trying to override OFFICIAL or SELF;
- prompt injection inside candidate text;
- sensitive-data redaction;
- rules-only fallback;
- backward-compatible CLI;
- `rules` mode never making a model call;
- simple writing task not invoking the model in automatic trigger logic.

### Evaluation cases

Expand from 10 to at least 30 product evaluations. Include:

- renovation, account appeal, hotel, inspection, contracts, local errands;
- ambiguous cases where market entry should be false;
- mixed routes;
- model/rule conflict;
- unknown slang with low confidence;
- prompt injection;
- credentials and off-platform payment;
- malformed model response;
- local-model timeout;
- non-Chinese inputs;
- user explicitly requesting official channels only.

### No live-provider dependency in CI

CI must use fake adapters and fixtures. Live LM Studio or hosted-provider tests are manual and skipped by default.

## 13. OpenMinis integration

OpenMinis should continue using direct CLI execution.

Recommended path after v0.2:

```bash
/root/Dreaminmaster-market-capability-router/.venv/bin/mcr \
  analyze @/tmp/mcr-user-request.txt \
  --mode hybrid
```

If no model configuration exists, OpenMinis should either use `--mode rules` or allow hybrid mode to fall back cleanly. It must not silently send data to an unknown remote provider.

## 14. Deliverables

A v0.2 pull request must include:

- adapter protocol;
- OpenAI-compatible adapter;
- fake adapter and deterministic fixtures;
- schemas and validator;
- redaction and prompt-injection checks;
- hybrid merge logic;
- CLI configuration;
- at least 30 product evaluation cases;
- updated Skill and OpenMinis integration docs;
- updated acceptance criteria, changelog, progress, decisions, and test report.

## 15. Non-goals

Do not implement in v0.2:

- screenshot OCR or image understanding;
- browser automation;
- direct marketplace scraping;
- automatic seller messaging;
- automatic payments or account login;
- vector databases;
- automatic promotion of model-generated slang into verified data;
- a graphical interface.

## 16. Release gate

v0.2 is accepted only when:

1. all v0.1.1 tests still pass;
2. new unit tests pass;
3. at least 30/30 product evaluations pass, or the documented threshold is at least 95% with no critical safety failure;
4. rules mode works with no network;
5. hybrid mode falls back on model failure;
6. critical risk and required OFFICIAL/SELF routes cannot be overridden;
7. no secret appears in logs, exceptions, fixtures, or committed files;
8. OpenMinis can run one rules-mode and one hybrid-mode smoke test;
9. `git status` is clean after validation;
10. the implementation report includes exact commands, commit SHA, test output, and any unresolved limitations.
